## 设计决策

### 清洗时机：OCR 后、LLM 提取前

```
截图 → PaddleOCR → 原始行 → _clean_text() → 评论清洗 → 合并文本 → DeepSeek
                                      ↑
                                 _remove_comment_markers()
```

在 `_clean_text` 的行过滤 + 合并之后，对所有文本执行一次正则扫描。此时文本已是连续段落，三种模式的匹配不受行边界影响。

### 三种正则模式

| 模式 | 正则 | 示例输入 | 输出 | 误伤风险 |
|------|------|----------|------|----------|
| Unicode 上标 | `[¹²³⁴⁵⁶⁷⁸⁹⁰]+` | `北伐¹²` | `北伐` | 零 |
| 方括号编号 | `\[\d+(,\d+)*\]` | `襄阳[1]` | `襄阳` | 零 |
| 句末标点+数字 | `([。！？…]\|\.{2,})\d{1,3}(?![万…])(?=[一-鿿])` | `。14相对应` | `。相对应` | 极低 |

### 句末标点模式的设计权衡

**为什么后向排除 `[万千百十余数两几]`**：
- "。5万铁骑" → 数字后接"万"，是正文数量词，保留
- "。10余人" → 数字后接"余"，是正文数量词，保留

**已知局限**：
- "。122两支" → "两"被排除规则保下来了，122 未被移除。宁可漏删也不误删正文。

### 为什么不在 LLM 侧处理

方案对比：
- OCR 侧正则清洗：零延迟、零成本、确定性行为、可测试
- LLM Prompt 规则：增加 token 消耗、结果不确定、每次调用都消耗成本

选择 OCR 侧正则清洗，因为评论数字的模式高度规则化，不需要语义理解。
