## Context

当前 `extractor.py` 中两个 System Prompt（`_SYSTEM_PROMPT` 和 `_TIMELINE_SYSTEM_PROMPT`）指导 LLM 从战役文本中提取地名（places），但未区分"独立地名"和"军队编制中的地名修饰语"。当文本出现"秦凤路大军"、"泾原路兵马"时，LLM 会将"秦凤路"、"泾原路"作为 places 提取，导致地图上出现虚假地点。

提取后的结果直接通过 Pydantic 校验，没有任何后处理过滤环节。

## Goals / Non-Goals

**Goals:**
- 在 Prompt 层面增加排除规则，让 LLM 理解军队编制名 ≠ 地名
- 在提取后增加后处理过滤函数，作为安全网二次校验
- Place 模型新增 `place_type` 字段，为后续地形可视化打基础

**Non-Goals:**
- 不引入新的外部依赖或 NER 库
- 不改变 API 接口签名（向后兼容）
- 不处理 OCR 层的文本清洗（那是另一个问题）

## Decisions

### 决策1：Prompt 规则 + 后处理双重保障

**选择**：同时在 Prompt 中增加规则 AND 在代码中增加后处理过滤函数。

**理由**：
- Prompt 规则是主力防线，LLM 能理解上下文语义，效果最好
- 后处理过滤是安全网，LLM 输出并非100%可控，正则/关键词后处理确保万无一失
- 两者互补，不增加额外延迟

**替代方案**：
- ❌ 仅 Prompt 规则：依赖 LLM 100% 遵守规则，不可靠
- ❌ 仅后处理过滤：正则无法覆盖所有语义变体（如"秦凤路各部"、"泾原兵马"等）

### 决策2：后处理采用上下文字段匹配

**选择**：利用 LLM 已返回的 `context` 字段（原文句子片段）做军队编制关键词匹配，而非对原文全文重新扫描。

**理由**：
- `context` 字段已包含地名出现的上下文，无需额外文本
- 匹配模式：`地名 + 军/大军/兵马/部队/将士/诸军/各部` 等军队后缀
- 复杂度低，无需引入 NLP 依赖

**替代方案**：
- ❌ 对原文全文做 NER：引入额外依赖和延迟
- ❌ 维护军队编制名黑名单：无法穷举，维护成本高

### 决策3：Place 模型新增可选 `place_type` 字段

**选择**：在 `Place` 模型中新增 `place_type: Optional[str]`，枚举值包括 `city`（城池）、`mountain_pass`（关隘）、`river`（河流）、`mountain`（山脉）、`region`（行政区/区域）、`battlefield`（战场）。

**理由**：
- 为后续「山川河流地形可视化」做准备，不同类型地名需要不同渲染方式
- 设为 Optional，当前 Prompt 逐步引导 LLM 输出该字段，不强制
- 向后兼容，旧数据不受影响

## Risks / Trade-offs

- [Prompt 规则可能被 LLM 忽略] → 后处理过滤作为兜底保证
- [后处理可能误杀合法地名] → 仅在 `context` 明确包含军队后缀时过滤；被过滤的地名记录日志供人工复核
- [`place_type` 字段增加 LLM 输出复杂度] → 设为 Optional，LLM 不确定时可不填，不影响现有流程
