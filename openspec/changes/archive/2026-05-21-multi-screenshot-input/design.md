## Context

当前系统仅支持单张截图OCR（`POST /api/ocr`），用户一次只能上传一张截图。实际场景中，一章网文通常需要3-5张截图才能完整覆盖。用户需重复"截图→上传→等待OCR→复制文本→粘贴到别处暂存"的流程，最后手动合并后再粘贴回输入框，体验割裂。

核心工作流不变：截图 → OCR → 清洗 → 提取。本次变更在OCR层和提取层之间插入"多图文本拼合"环节。

现有 `_clean_text()` 已处理单图层面的UI噪声清洗，拼合层只关心跨图的重复文本去除。

## Goals / Non-Goals

**Goals:**
- 支持一次上传多张截图（多文件选择 / 多次粘贴追加）
- 后端批量OCR后自动执行相邻图文去重和拼接
- 通过SSE实时反馈每张图的OCR进度
- 拼接结果供用户确认后再送入提取管道
- 单图场景的原有API和流程不受影响

**Non-Goals:**
- 不实现拖动排序（首版按上传/粘贴顺序排列）
- 不做图片级别的去重（如同一张图传了两次），只做文本级别
- 不改变 LLM 提取管道的分段策略（本次不做超长文本分段提取）
- 不去改 `POST /api/ocr` 的行为（保持单图接口不变）

## Decisions

### 1. 新增 `POST /api/ocr/batch` 接口

新增批量接口而非修改现有接口。原因：
- 单图 `POST /api/ocr` 已有调用方，改动有破坏风险
- 批量接口的请求格式（multipart多文件）和响应格式（SSE流）与单图接口完全不同
- 后端代码可复用 `ocr_main()` 作为单图处理单元

### 2. 去重算法：前后缀最长公共子串

相邻截图的文本在首尾可能有重叠（用户截图时多截了一两行保证不漏）。采用滑动窗口最长公共子串匹配：

```
图1文本: "...刘备率军西进，于建安二十四年抵达汉中。"
图2文本: "于建安二十四年抵达汉中。诸葛亮在..."
                              ^^^^^^^^^^^^ 重叠部分
```

算法：在图1尾部取滑动窗口（默认取末尾200字符），在图2头部查找最长匹配子串，找到后去除图2的重复前缀。

选择此算法而非编辑距离或语义匹配的原因：
- 截图重叠是共享相同文本（精确匹配场景），不是语义近似
- LCS匹配是O(n*m)，但窗口限定在200字符内性能可接受
- 不需要额外依赖（纯Python实现）

备选方案及舍弃原因：
- LLM判断去重：延迟高、成本大、过度设计
- SimHash近似去重：截图重叠是精确文本匹配，不需要模糊哈希
- difflib.SequenceMatcher：可行但过重，手动实现LCS窗口匹配更可控

### 3. 去重失败时的降级策略

如果两张图的文本完全没有重叠（用户精确对齐截图），直接拼接，不做去重。用户在确认环节可以手动修正。

### 4. SSE流式进度反馈

前端 POST `/api/ocr/batch` 后，后端通过 SSE 推送每张图的OCR进度：

```
event: progress
data: {"current": 2, "total": 5, "status": "ocr_done", "char_count": 320}

event: merge
data: {"original_chars": 1580, "merged_chars": 1420, "removed_dup": 160}

event: complete
data: {"text": "...完整合并文本..."}
```

前端据此展示进度条和每张图的状态。

### 5. 前端截图管理

在现有上传区域增加：
- `input[multiple]` 支持多文件选择
- 粘贴事件支持追加模式（已有Ctrl+V粘贴的基础上，允许多次粘贴）
- 缩略图列表展示已添加的截图（按添加顺序编号）
- 每张截图旁有删除按钮
- "开始识别"按钮触发批量处理

## Risks / Trade-offs

- **去重误删**: 如果正文中恰好有重复段落（如反复出现的诗句），可能被去重算法误判为重叠 → 用户确认环节可手动恢复；去重仅在相邻图的首尾窗口中进行，大幅降低误删概率
- **超长文本**: 5张截图每张提取~300字，合并后约1500字，在DeepSeek token限制内；但如果用户截了10张，可能超出 → 首版不做分段处理，后续按需迭代
- **OCR累积延迟**: 5张图串行OCR可能需要10-20秒 → SSE进度反馈让用户感知进度，避免以为卡死

## Open Questions

- 是否需要支持用户手动调整截图顺序（拖动排序）？ → 首版按添加顺序，后续按需求决定
- 如果某一单张截图OCR失败（如图片模糊），整个batch应该失败还是跳过继续？ → 首版设计为：某张失败则整体失败并提示第几张出问题，让用户重新截该张
