## Context

当前系统从战役文本中提取的是静态地理信息（地名坐标、行军路线），输出 `CampaignExtract/CampaignTimeline` 包含 factions、places、routes、events。但历史小说中的战役段落核心是「战术动态」——部队转向、侧翼包抄、骑兵冲击、全线崩溃等动作发生在同一地理区域，现有地图渲染只有静止的地名标记和路线线条，无法表达战术层面的空间叙事。

前端为单文件 vanilla JS（~900 行），使用 MapLibre GL JS 渲染。时间线系统已支持按 step 过滤地名和路线的显隐。本次设计在现有架构上扩展，不引入新框架。

## Goals / Non-Goals

**Goals:**
- 新增 `ForceUnit` 和 `UnitState` 数据模型，使部队成为一等实体
- LLM 提取部队实体并追踪跨事件状态变化（复用现有 timeline 模式的一次 API 调用）
- 地图上以**块状箭头**（宽箭身+方向箭尖）展示部队，统一表达位置、进攻方向和兵力规模，状态通过箭头形态区分
- 实现四种战术动作动画类型：路径（move）、显隐（appear/disappear）、属性渐变（status transition）、生长（route growth）
- 时间线步进时部队标记自动更新位置和样式
- 多个部队在同一地名时自动做位置偏移，避免标记重叠

**Non-Goals:**
- 不引入前端框架（保持 vanilla JS）
- 不新增独立 API 端点（扩展现有 `/api/extract` SSE 流）
- 不做平滑插值动画（部队位置按 step 快照切换，不做帧间过渡）
- 不使用 NATO/APP-6 军用符号（与古代战役语境不符）
- 不要求 LLM 输出精确经纬度（部队位置以地名为锚点）
- 不支持手动创建/编辑部队（MVP 仅 AI 自动提取）

## Decisions

### D1: 部队位置以地名为锚点，不做独立地理坐标

**选择**：`UnitState.location` 存储对 `places` 数组中地名的引用（字符串匹配），前端渲染时以该地名的坐标为基础做偏移。

**备选方案**：让 LLM 独立输出每个部队状态的经纬度。拒绝理由：(1) 小说文本只描述相对位置（「塬地南侧」「贴着塬底」），LLM 无法凭空生成精确坐标；(2) 独立坐标会与 CHGIS 地名标记产生位置矛盾。

**影响**：需要在前端实现「同地多部队偏移算法」——以地名坐标为圆心，按部队数量做环形分布。

### D2: 部队状态快照模型，不做连续运动

**选择**：部队在时间线的每个 step 有一个 `UnitState` 快照（位置+状态+描述），步进时瞬间切换，不做帧间平滑过渡。

**备选方案**：用 `requestAnimationFrame` 做部队标记沿路径平滑移动。拒绝理由：(1) vanilla JS 实现复杂，易出 bug；(2) 文本阅读场景下用户按步查看，快照切换已足够；(3) 平滑动画需要的中间坐标数据 LLM 无法提供。

**影响**：动画模型简化为「显隐变化 + 属性变化」，不包含连续路径动画。未来如需平滑动画可在此模型上叠加插值层。

### D3: 块状箭头统一表达位置、方向和规模

**选择**：部队标记采用历史战役示意图经典的「块状箭头」设计——箭身宽度表达进攻正面/兵力规模，箭尖方向表达进攻方向。箭身即模糊区域，不需额外渲染半透明椭圆。

**灵感来源**：历史战役示意图（孟良崮、渡江战役、南京战役等）中的块状宽箭头，以及 GeoBlobs 的位置不确定性表达理念。

**箭头结构**：
```
╔══════════════╗
║   焦文通部    ╲        ← 箭身宽度 ∝ 进攻正面/兵力规模
║   数千人      ╱
╚══════════════╝▶ 娄室中军  ← 箭尖指向进攻方向
     ↑
  阵营色填充 (宋蓝/金红)
```

**影响**：前端需实现 Canvas/SVG 箭头绘制逻辑，生成 GeoJSON Polygon feature 作为 MapLibre fill 图层。箭身长度、宽度、箭尖比例需根据 scale 级别自适应。不再需要独立的模糊区域图层（箭头体本身就是位置-方向-规模的三合一表达）。

### D4: 复用现有时间线过滤机制

**选择**：每个部队标记 GeoJSON feature 携带 `step` 属性，使用与现有地名/路线相同的 MapLibre filter 模式（`['<=', ['get', 'step'], currentStep]`）。状态为 `destroyed` 的部队在 step 到达时显示为破碎样式而非隐藏。

**备选方案**：新建独立的部队图层管理系统。拒绝理由：与现有模式不一致，增加维护成本。

### D5: LLM 以名称字符串维持部队身份

**选择**：`UnitState` 通过 `unit_name` 字符串字段关联到 `ForceUnit.name`。LLM prompt 中要求部队名称在整个 JSON 输出中保持一致。后处理校验发现不匹配时，按名称相似度自动纠正或丢弃。

**备选方案**：让 LLM 输出显式的 unit_id（数字索引）。拒绝理由：LLM 对数字 ID 的跨事件一致性更差，名称字符串是 LLM 更自然的引用方式。

### D6: 块状箭头形态区分状态，不只是颜色

**选择**：部队状态通过箭头**形态**区分（参考历史战役图中进攻=实心宽箭、撤退=细虚线箭的做法），颜色辅以阵营标识。这是与原有「纯色块」方案的关键差异。

| 状态 | 箭头形态 | 颜色 | 说明 |
|------|---------|------|------|
| 待命/列阵 (deploying) | 实心矩形，无箭尖 | 阵营色 60% 透明 | 尚未形成进攻方向，不渲染箭尖 |
| 进军/进攻 (marching) | 标准块状箭头，箭尖完整 | 阵营色实心 | 核心形态，箭尖指向进攻目标 |
| 交战 (engaging) | 块状箭头 + 红色闪烁边框 | 阵营色实心，红色粗边框 | 正在对抗，视觉上"在燃烧" |
| 撤退 (retreating) | 细虚线反向箭头，箭尖缩小 | 阵营色 30% 透明 | 与历史地图「细虚线=撤退」一致 |
| 溃散 (routing) | 箭头碎裂为散点 | 灰色半透明 | 比撤退更退一步，表示部队瓦解 |

**备选方案**：使用 milsymbol 库的 NATO APP-6 符号。拒绝理由：古代中国战役使用现代军事符号会产生时代违和感；且北约军标体系没有「块状宽箭头」概念，它本身就是中国/苏联军事制图学的经典表达。

## Risks / Trade-offs

- **[风险] LLM 跨事件部队名称一致性差** → 缓解：后处理用字符串相似度（编辑距离）自动合并名称变体；prompt 中明确要求「部队名称在整个 JSON 输出中保持一致，不要在同一部队的不同事件中使用别名」
- **[风险] 同名地下多个块状箭头重叠** → 缓解：同地多部队的箭头沿进攻方向做平行错位（类似多条车道），间距按箭头宽度×1.2；进攻方向不同的部队以地名为起点向各自方向辐射
- **[风险] 块状箭头在极小 scale 下不可读** → 缓解：strategic 级别（zoom≤6）简化为细线箭头，battle 级别（zoom≤10）为标准块状箭头，tactical 级别（zoom≤14）展示完整箭头+标签
- **[风险] vanilla JS 状态管理复杂** → 缓解：将部队相关逻辑封装为独立模块 `unitLayerManager`，通过全局状态对象与现有代码解耦
- **[取舍] 不做平滑动画损失视觉冲击力** → 可接受：文本阅读场景以信息清晰度为优先，快照切换已满足需求；后续可叠加插值层

## Open Questions

- 块状箭头的箭身-箭尖比例：参考历史图（1:3~1:4），实际渲染时是否需根据屏幕像素动态调整？
- 箭头在 MapLibre 中的实现方式：SVG icon 叠加，还是 GeoJSON Polygon fill-extrusion？前者简单但缩放时比例固定，后者更灵活但实现复杂。初步建议 SVG icon 方案（MVP 友好）
- 部队兵力描述（「数千」「满员一千骑」）是否需要在箭头宽度上做视觉区分？初步建议暂不做，避免信息过载
